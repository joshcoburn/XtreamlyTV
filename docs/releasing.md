# Release process

XtreamlyTV uses semantic versioning while the Android target remains a preview. A tag such as `v0.4.0` triggers the release workflow.

## 1. Update versions

Update all of the following:

```text
VERSION
package.json
packages/core-web/package.json
apps/webos/app/appinfo.json
apps/webos/app/js/app.js (APP_VERSION)
apps/android-tv/app/build.gradle.kts (versionName/versionCode)
CHANGELOG.md
```

## 2. Regenerate platform files

```bash
npm run prepare:apps
```

## 3. Run checks

```bash
npm run check:webos
npm run test:core
python3 apps/webos/tests/browser_smoke.py
npm run build:webos
cd apps/android-tv && ./gradlew :app:testDebugUnitTest :app:assembleDebug
```

## 4. Commit and tag

```bash
git add .
git commit -m "Release v0.4.0"
git tag -a v0.4.0 -m "XtreamlyTV v0.4.0"
git push origin main --follow-tags
```

## 5. GitHub release

The tag workflow builds and attaches:

- webOS IPK
- Android debug APK for development testing
- checksums
- source archives generated by GitHub

Public production Android distribution should use a signed release APK/AAB. Never commit the signing keystore or passwords.

## Release channels

- `v0.x.y`: development/beta releases
- GitHub prerelease: Android feature preview or potentially disruptive playback changes
- Stable: only after both platform checklists and physical-device testing pass
